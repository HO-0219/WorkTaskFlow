# 2026-07-21 개발 로그

## 작업 단계

- 단계: 개발 준비 및 기반 구조 정리
- 상태: 진행
- 기능 개발 여부: 신규 업무 도메인 기능 개발 전

오늘 작업은 기존 인증 스타터의 동작을 유지하면서 앞으로의 개발 기반과 문서를 정리한 것이다. 인증 기능, 파일 업로드, S3 기능을 새로 완성한 것으로 기록하지 않는다.

## 오늘의 목표

- 업무 도메인을 추가하기 전에 백엔드 패키지 스캔과 책임 경계를 정리한다.
- 로컬 파일 저장과 향후 S3 전환 원칙을 문서로 확정한다.
- 프로젝트의 전체 개발 순서와 날짜별 개발 로그 형식을 마련한다.
- Phase 0의 사이트맵, 사용자 흐름, 와이어프레임, 디자인·권한·데이터·API 기준선을 작성한다.
- Flyway로 인증 DB 스키마를 재현 가능하게 만든다.

## 완료한 작업

### 백엔드 기반 구조

- Spring Boot 시작 클래스를 `com.teamproject.TeamProjectApplication`으로 이동했다.
- Spring 컴포넌트, JPA Entity, Repository의 최상위 스캔 기준을 `com.teamproject`로 변경했다.
- 기존 `com.teamproject.auth` 코드를 다음 책임으로 분리했다.
  - `user`: User 데이터와 계정 생명주기
  - `authentication`: 가입·로그인·복구·세션·OAuth
  - `authorization`: 공개/보호 API와 향후 역할 정책
  - `jwt`: Access JWT 생성·검증과 Bearer 인증 필터
  - `common`: 공통 예외, API 오류, 헬스체크
- 테스트 클래스도 새 패키지와 시작 클래스를 사용하도록 이동했다.
- 테스트 설정에 `spring.jpa.open-in-view=false`를 추가했다.

### 파일 저장 설계

- `backend/uploads/.gitignore`를 만들어 실제 업로드 파일은 Git에서 제외하도록 했다.
- 로컬과 S3가 공통으로 사용할 상대 `storageKey` 형식을 문서화했다.

```text
groups/group-{groupId}/{yyyy}/{MM}/{UTC timestamp}_member-{groupMemberId}_{uuid}.{ext}
```

- 변경·중복 가능한 groupName과 멤버 닉네임은 실제 저장 경로에 사용하지 않기로 했다.
- 저장 단계는 알파의 `backend/uploads`, 베타의 서버 영구 볼륨, 발표 후 출시 후보의 S3 순서로 결정했다.
- 파일 업로드, LocalFileStorage, S3FileStorage 자체는 아직 구현하지 않았다.

### 문서

- `프로젝트_기초구조_설명.txt`에 실제 패키지 구조와 Resource/저장 정책을 반영했다.

- `docs/spec`에 팀장/팀원 권한 매트릭스, 초기 ERD, 인증 계약과 향후 그룹·업무 API 초안을 작성했다.

### DB 마이그레이션

- Flyway Core와 MySQL 모듈을 추가했다.
- `V1__create_authentication_schema.sql`에 현재 인증 테이블 4개의 생성 기준을 작성했다.
- 애플리케이션 기본 DDL 정책을 `update`에서 `validate`로 변경했다.
- 기존 로컬 DB 도입을 위한 `baseline-on-migrate` 정책과 기준선 등록 후 해제 방법을 README에 기록했다.
- H2의 MySQL 호환 모드에서 V1 적용 → Hibernate validate → 인증 흐름을 연속 검증했다.
- Docker MySQL 8.4의 빈 `teamProject` 스키마에 Flyway V1을 실제 적용하고 재기동 시 version 1 검증을 확인했다.
- 로컬 `.env`의 Hibernate DDL 설정도 `update`에서 `validate`로 변경했다.

### 인증 로그 보안

- 실제 HTTP 스모크 로그에서 DEBUG 요청 로깅이 비밀번호·인증번호·access token을 DTO 문자열로 출력할 수 있음을 확인했다.
- 인증 요청·응답 DTO의 `toString()`이 비밀번호, 인증번호, 재설정 토큰, access token, 개인정보를 노출하지 않도록 마스킹했다.
- Spring Web와 Hibernate SQL의 애플리케이션 기본 로그 수준을 INFO로 제한했다.
- 민감 DTO 문자열에 원문이 포함되지 않는 회귀 테스트 2개를 추가했다.

## 결정 사항

- User Entity의 소유자는 `user`로 한정한다.
- Refresh Token은 JWT가 아닌 난수 기반 세션 토큰이므로 `authentication/domain/token`에 둔다.
- Access Token의 JWT 구현과 Bearer 인증 필터만 `jwt`에 둔다.
- 기존 `/api/v1/auth/**` 경로, DB 테이블, JWT Claim, Refresh Cookie 계약은 패키지 이동 전후 동일하게 유지한다.
- 신규 기능은 디자인과 권한 흐름을 먼저 확정한 뒤 세로 기능 단위로 개발한다.
- AI 리포트는 별도 담당 트랙으로 분리하고 핵심 업무와 기본 통계가 AI에 의존하지 않게 한다.
- S3 전환은 발표 후 운영 유사 스테이징에서 검증한 뒤 진행한다.
- 화면 기준선은 인디고 계열의 업무 집중형 UI, PC 사이드바와 모바일 하단 탐색으로 정했다.
- 업무 권한은 인증 상태, 시스템 권한, 그룹 멤버십 권한을 분리해 검사한다.
- DB 스키마 변경은 Hibernate 자동 수정 대신 새 Flyway migration으로만 수행한다.

## 영향 범위

- API: 변경 없음
- DB 스키마: 새 환경용 인증 스키마 V1 추가, 현재 엔티티 테이블 구조 자체는 유지
- 환경변수: Hibernate 기본값 `validate`, Flyway 활성화·baseline 설정 추가
- 인증 호환성: 기존 JWT Claim과 Refresh Cookie 유지
- 프런트 기능: 변경 없음
- 파일 업로드 기능: 미구현, 디렉터리와 정책만 준비
- 배포: 실행 JAR의 시작 클래스가 `com.teamproject.TeamProjectApplication`으로 변경

## 이슈 및 주의사항

- 현재 `AuthService`, `AuthController`, `AuthDtos`는 패키지 경계만 이동했으며 signup, login, recovery, session, profile, withdrawal 단위로 아직 분리하지 않았다.
- Flyway 방식은 확정했으며 Phase 1 사용자 필드 변경부터 V2 migration을 추가해야 한다.
- 현재 자동 테스트는 인증 서비스 흐름과 컨텍스트 로드 중심이라 HTTP API, 쿠키 속성, JWT 변조·만료, 401/403 계약 테스트를 보강해야 한다.
- Spring Boot 3.3.5가 관리하는 Flyway 10.10.0은 MySQL 8.4가 해당 버전에서 공식 검증된 범위보다 새 버전이라는 경고를 출력한다. 현재 V1 적용과 재기동은 성공했지만 의존성 업그레이드 때 다시 확인해야 한다.
- 현재 변경 사항은 커밋하거나 원격 저장소에 푸시하지 않았다.
