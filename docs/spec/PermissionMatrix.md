# 권한 매트릭스

상태: 로컬 알파 서버 인가 구현 기준

## 권한 축

권한은 세 축을 혼합하지 않는다.

1. 인증 상태: 비로그인 / 로그인
2. 시스템 권한: `USER` / `ADMIN` — Java `User.systemRole`, 기존 DB/JWT `role` 계약 유지
3. 그룹 권한: 비멤버 / `MEMBER` / `LEADER` — `GroupMember.role`

대부분의 업무 API는 시스템 `USER/ADMIN`보다 활성 그룹 멤버십을 먼저 검사한다. `ADMIN`도 그룹 자료를 자동으로 열람하지 않으며, 별도 운영자 기능이 만들어지기 전에는 일반 사용자와 같은 그룹 권한을 적용한다.

## 서버 인가 순서

```text
JWT 인증
→ 대상 자원 존재 확인
→ 자원의 groupId 확인
→ 활성 GroupMember 존재 확인
→ 그룹 역할 확인
→ 소유자/요청자/담당자 조건 확인
→ 현재 상태·버전·정책 확인
→ 변경 수행
```

- 다른 그룹 자원에 접근하면 정보 노출을 줄이기 위해 API별로 403 또는 404 중 하나를 일관되게 사용한다. 기본값은 존재를 이미 아는 그룹 멤버의 권한 부족은 403, 비멤버의 자원 직접 접근은 404다.
- 프런트 버튼 숨김은 편의 기능일 뿐 서버 검사를 대신하지 않는다.
- 모든 권한 규칙은 정상·비멤버·역할 부족·탈퇴 멤버 테스트로 고정한다.
