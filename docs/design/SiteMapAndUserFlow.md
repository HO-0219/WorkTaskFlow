# 사이트맵과 핵심 사용자 흐름

상태: 로컬 알파 구현 경로 기준

## 사이트맵

```text
비로그인
├── /login                         로그인
├── /signup                        회원가입·이메일 인증
├── /find-username                 아이디 찾기
├── /forgot-password               비밀번호 재설정 요청
├── /reset-password                비밀번호 재설정
├── /oauth/callback                소셜 로그인 완료 처리
└── /group-invitations/accept      로그인 후 그룹 초대 수락

로그인
├── /                              개인 통합 대시보드
├── /groups                        참여 그룹 목록·그룹 생성
├── /groups/:groupId               그룹 설정·멤버·초대 관리
├── /groups/:groupId/tasks         그룹 업무 목록·등록
├── /groups/:groupId/dashboard     그룹 대시보드
├── /tasks/:taskId                 업무 상세·상태·체크리스트·댓글
├── /calendar                      개인·참여 그룹 통합 캘린더
├── /notifications                 알림 센터
├── /profile                       프로필
└── /account                       비밀번호 변경·탈퇴
```

현재 로컬 알파는 각 화면 상단의 링크와 홈 대시보드 카드를 전역 탐색으로 사용한다. PC는 다열 카드·목록, 모바일은 단일 열로 재배치한다. 좌측 사이드바·하단 탭·상세 패널은 후속 셸 개선 후보이며 현재 경로 계약에 포함하지 않는다.

URL은 공유와 새로고침이 가능해야 한다. 그룹 화면은 항상 `groupId`, 업무 화면은 항상 `taskId`를 URL에 포함한다. 표시 이름은 경로나 권한 판단에 사용하지 않는다.

## 핵심 흐름

### 1. 가입과 로그인

```text
로그인 → 회원가입 → 이메일 인증번호 발송 → 인증번호 확인
→ 계정 생성 → PERSONAL 그룹 자동 생성 → 로그인 → 개인 대시보드
```

- 홈은 저장된 access token 또는 refresh cookie로 인증 상태를 확인하며, 인증 화면은 직접 접근도 허용한다.
- 홈 진입 시 access token이 없으면 refresh cookie로 한 번 재발급한다.
- 다른 화면에서 access token이 만료되면 현재 로컬 알파는 오류 안내 후 재로그인이 필요하다. 전역 자동 refresh는 동결 이후 개선 항목이다.
- 아이디 찾기와 비밀번호 찾기는 계정 존재 여부를 응답에서 노출하지 않는다.

### 2. 개인 공간 진입

```text
로그인 → 기본 PERSONAL 그룹 확인
→ 개인 대시보드 → 개인 업무 등록 → 즉시 TODO → 수행·완료
```

- 개인 그룹은 별도 생성·초대·탈퇴 기능이 없다.
- 개인 업무는 팀장 승인 없이 생성 즉시 `TODO`가 된다.

### 3. 팀 그룹 생성·초대·가입

```text
그룹 목록 → 팀 그룹 생성 → 생성자는 LEADER
→ 멤버 초대 → 초대 수락 → MEMBER로 가입 → 그룹 대시보드
```

- 초대가 만료·취소·이미 처리된 경우 원인별 안내와 그룹 목록 이동을 제공한다.
- 그룹 진입 때 활성 멤버십이 없으면 서버의 `404 GROUP_NOT_FOUND` 안전 오류를 표시하고 데이터를 렌더링하지 않는다.

### 4. 업무 제안부터 완료

팀원이 등록하는 경우:

```text
업무 등록(REQUESTED) → 팀장 알림 → 수락 또는 반려
→ 수락·담당자 할당(TODO) → 담당자 알림 → 시작(IN_PROGRESS)
→ 필요 시 보류(ON_HOLD)·재개 → 완료(COMPLETED) → 팀장 알림
```

TEAM 그룹은 등록자의 역할과 관계없이 `REQUESTED`로 시작한다. LEADER가 등록한 경우에도 승인한 뒤 담당자를 지정해야 `TODO` 수행 흐름으로 넘어간다.

- 반려·보류·취소는 사유 입력을 완료해야 확정된다.
- 사용 가능한 상태 변경 버튼은 역할과 현재 상태를 모두 만족할 때만 보이지만, 최종 검사는 서버가 수행한다.
- 상태 변경 성공 후 목록 요약, 상세 이력, 알림 숫자를 함께 갱신한다.

### 5. 알림에서 대상 화면 이동

```text
알림 링크 → 최신순 목록 → 알림 선택 → 읽음 처리
→ taskId가 있으면 업무 상세, 없으면 홈으로 이동
```

- 권한을 잃었거나 대상이 삭제된 경우 서버의 안전 오류를 표시하며, 알림 자체의 읽음 처리는 수신자 본인이 계속 할 수 있다.
- 동일한 이벤트로 담당자와 멘션 대상이 겹치면 알림은 한 번만 생성한다.

## 반응형 기준

- `0~767px`: 모바일 단일 열, 상단 링크를 줄바꿈하고 상세 화면을 전체 폭으로 표시
- `768~1199px`: 태블릿/소형 PC, 화면별 다열 레이아웃을 가용 폭에 맞게 축소
- `1200px 이상`: 화면별 카드·목록의 최대 폭과 다열 레이아웃 사용
- 지원 최소 너비는 320px이며 핵심 동작의 터치 영역은 최소 44px이다.

## 화면 접근 원칙

- 인증 필요 화면 접근 실패: 로그인 후 원래 경로로 복귀한다.
- 그룹 비멤버: 403 전용 상태를 표시한다.
- 존재하지 않는 자원: 404 전용 상태를 표시한다.
- 기능 플래그가 꺼진 Phase 7 기능은 탐색 메뉴 자체를 노출하지 않는다.
