# 로컬 알파 테스트 체크리스트

기준 환경은 Java 21, Node.js 20+, 로컬 MySQL 8.x이다. Docker·AWS·S3·CI/CD는 이 체크리스트의 범위가 아니다.

## 1. 실행

- [ ] MySQL의 `teamProject` DB가 실행 중이고 루트 `.env`의 DB 정보와 일치한다.
- [ ] `cd backend && ./mvnw spring-boot:run` 후 Flyway schema version 12와 8081 기동을 확인한다.
- [ ] `cd frontend && npm run dev` 후 `http://localhost:5174`에 접속한다.
- [ ] `http://localhost:8081/api/v1/health`가 정상 응답한다.

## 2. 인증과 계정

- [ ] 이메일 인증번호를 요청하고 백엔드 `[LOCAL MAIL]` 로그의 6자리 코드로 가입한다.
- [ ] 가입 후 로그인·새로고침(refresh cookie)·로그아웃이 동작한다.
- [ ] 아이디 찾기와 비밀번호 재설정 메일을 `[LOCAL MAIL]` 로그로 확인한다.
- [ ] 프로필의 닉네임·전화번호·이미지 URL을 수정하고 새로고침 후에도 유지되는지 확인한다.
- [ ] 비밀번호 변경 후 이전 refresh 세션이 폐기되고 새 비밀번호로만 로그인되는지 확인한다.
- [ ] 테스트 계정 탐퇴 후 로그인과 보호 화면 접근이 차단되는지 확인한다.

## 3. 그룹

- [ ] 신규 가입 계정으로 `/groups`에 진입했을 때 PERSONAL 그룹이 정확히 하나 보인다.
- [ ] 팀 이름과 설명을 입력해 TEAM 그룹을 생성하면 목록에 즉시 추가되고 역할이 팀장으로 표시된다.
- [ ] 새로고침 후에도 개인·팀 그룹이 MySQL에서 다시 조회된다.
- [ ] 그룹 카드를 눌러 상세 화면에 진입하고 이름·설명·시간대·대시보드 공개 범위를 저장한다.
- [ ] PERSONAL 그룹에서는 대시보드 공개 범위를 변경하는 입력이 노출되지 않는다.
- [ ] TEAM 그룹에서 가입된 다른 계정의 이메일로 초대하고 백엔드 `[LOCAL MAIL]` 로그에서 수락 링크를 확인한다.
- [ ] 초대 대상 계정으로 로그인한 뒤 링크를 열어 수락하고 그룹·멤버 목록에 추가되는지 확인한다.
- [ ] 다른 이메일 계정으로 수락하면 거부되고, 정상 수락한 링크를 다시 사용해도 거부된다.
- [ ] LEADER가 대기 중인 초대를 취소하면 해당 링크로 수락할 수 없다.
- [ ] 팀장이 팀원을 LEADER로 승격하고 자신을 MEMBER로 변경해 역할 위임이 동작하는지 확인한다.
- [ ] 마지막 LEADER의 역할 변경·탈퇴 시도가 차단되는지 확인한다.
- [ ] MEMBER가 직접 탈퇴하면 그룹 목록과 상세에 더 이상 접근할 수 없는지 확인한다.
- [ ] LEADER가 MEMBER를 내보내면 대상 계정의 그룹 접근이 즉시 차단되는지 확인한다.
- [ ] 탈퇴·제거된 계정을 다시 초대하면 기존 멤버십 이력을 재사용해 MEMBER로 참여하는지 확인한다.
- [ ] 로그아웃 상태로 `/groups`에 접근하면 로그인 화면으로 이동한다.

## 4. 업무 등록·조회

- [ ] PERSONAL 그룹에서 업무를 등록하면 즉시 `할 일`이며 요청 멤버가 담당자로 표시된다.
- [ ] TEAM 그룹에서 업무를 등록하면 `승인 대기`이며 담당자는 미지정으로 표시된다.
- [ ] 우선순위와 마감일을 입력한 업무가 목록·상세와 새로고침 후 동일하게 보인다.
- [ ] 지난 마감일의 미종료 업무에 `지연` 배지가 표시된다.
- [ ] 다른 그룹 계정은 업무 목록·상세 URL에 직접 접근해도 내용을 볼 수 없다.
- [ ] TEAM 업무를 팀원 계정으로 제안하면 팀원에게 승인·반려 버튼이 보이지 않고 팀장에게만 보인다.
- [ ] 팀장이 요청을 승인하고 활성 멤버를 담당자로 지정하면 상태가 `할 일`로 바뀐다.
- [ ] 담당자가 아닌 계정에는 시작·보류·완료 버튼이 보이지 않고 직접 API를 호출해도 거부된다.
- [ ] 담당자가 `시작 → 보류(사유) → 재개 → 완료`하면 상태·시작일·완료일·이력이 즉시 갱신된다.
- [ ] 반려·보류·취소에서 빈 사유가 차단되고 입력한 사유가 상세와 상태 이력에 남는다.
- [ ] 같은 업무를 두 탭에서 연 뒤 한 탭에서 변경하면 다른 탭의 오래된 변경 요청이 409로 차단된다.
- [ ] 요청자가 TEAM 승인 대기 업무의 제목·설명·우선순위·마감일을 수정하고 팀장이 수정 결과를 조회할 수 있다.
- [ ] TEAM 업무 승인 후에는 내용 수정 버튼이 사라지고 직접 수정 API도 차단된다.
- [ ] PERSONAL 미종료 업무는 내용을 수정할 수 있고 취소·완료 후에는 수정할 수 없다.
- [ ] 빈 설명과 마감일 삭제가 저장되고 새로고침 후에도 그대로 유지된다.

## 5. 체크리스트

- [ ] 항목이 없는 업무 상세에 `체크리스트 없음`이 표시되고 잘못된 `0%`가 표시되지 않는다.
- [ ] 담당자 또는 팀장으로 항목을 추가·수정·완료·미완료·삭제하면 새로고침 후에도 유지된다.
- [ ] 완료 수와 전체 수, 진행률 막대가 `0% → 일부 완료 → 100%`에 맞게 갱신된다.
- [ ] 일반 활성 멤버는 체크리스트를 볼 수 있지만 변경 입력은 보이지 않고 직접 API 호출도 거부된다.
- [ ] 완료·반려·취소된 업무에서는 담당자와 팀장도 체크리스트를 변경할 수 없다.
- [ ] 같은 항목을 두 탭에서 변경하면 오래된 탭의 수정·삭제가 409로 차단된다.
- [ ] 다른 그룹 계정은 체크리스트 URL을 직접 호출해도 내용을 볼 수 없다.

## 6. 댓글

- [ ] 활성 멤버가 댓글을 등록하면 닉네임·작성 시각과 함께 다른 멤버 화면에도 표시된다.
- [ ] 작성자에게만 수정·삭제 버튼이 보이고 팀장을 포함한 다른 멤버의 직접 API 요청은 거부된다.
- [ ] 댓글 수정 후 `수정됨`이 표시되고 새로고침 후에도 내용이 유지된다.
- [ ] 삭제 후 원문 대신 `삭제된 댓글입니다.`가 표시되며 작성자·작성 시각은 유지된다.
- [ ] 완료·반려·취소 업무에서도 활성 멤버가 후속 댓글을 작성할 수 있다.
- [ ] 같은 댓글을 두 탭에서 변경하면 오래된 탭의 수정·삭제가 409로 차단된다.
- [ ] HTML 태그나 스크립트 형태의 댓글이 실행되지 않고 텍스트로 표시된다.
- [ ] 그룹 외 계정과 탈퇴·제거 멤버는 댓글 목록·작성 API에 접근할 수 없다.
- [ ] 댓글 등록·수정 화면에서 여러 활성 멤버를 선택하면 `@닉네임` 배지가 중복 없이 표시된다.
- [ ] 댓글 수정에서 멘션을 추가·제거하면 새 목록으로 교체되고 새로고침 후 유지된다.
- [ ] 다른 그룹 또는 탈퇴·제거 멤버 ID를 직접 보내면 댓글과 멘션 저장이 함께 롤백된다.
- [ ] 삭제된 댓글에서는 기존 멘션 대상이 화면과 API 응답에 노출되지 않는다.

## 7. 알림

- [ ] 팀원이 TEAM 업무를 요청하면 팀장 알림 센터에 승인 대기 알림이 한 건 표시된다.
- [ ] 팀장이 담당자를 지정하면 담당자에게 알림이 표시되고 행위자 본인에게는 알림이 생기지 않는다.
- [ ] 댓글 담당자와 멘션 대상이 같아도 같은 댓글 알림이 한 건만 표시된다.
- [ ] 댓글 수정으로 새 멤버를 멘션하면 새 대상만 알림을 받고 기존 대상에게 반복 알림이 생기지 않는다.
- [ ] 알림을 선택하면 읽음 처리 후 해당 업무 상세로 이동한다.
- [ ] `모두 읽음` 후 읽지 않음 개수가 0이 되고 새로고침 후에도 유지된다.
- [ ] 알림이 20개를 넘으면 `이전 알림 더 보기`가 중복 없이 이어진다.
- [ ] 다른 계정의 알림 ID를 직접 읽음 처리하면 404로 거부된다.
- [ ] 새 알림 발생 후 최대 30초 안에 홈과 알림 센터의 읽지 않음 개수가 갱신된다.

## 8. 캘린더

- [ ] 개인 그룹에서 일정·회의·휴가·할 일 유형을 등록하고 수정·삭제하면 새로고침 후 동일하게 반영된다.
- [ ] TEAM MEMBER는 일정을 조회하지만 등록·수정·삭제 입력은 보이지 않고 직접 API 요청도 거부된다.
- [ ] 전체 필터에서 ACTIVE로 참여 중인 개인·팀 그룹 일정이 시간순으로 함께 표시된다.
- [ ] 그룹 필터와 이전·현재·다음 달 이동이 URL 및 조회 결과와 일치한다.
- [ ] 업무에 마감일을 추가·변경·삭제하면 캘린더 마감 항목이 복제 없이 즉시 추가·이동·제거된다.
- [ ] 업무 마감 항목의 `업무 보기`가 해당 업무 상세로 이동한다.
- [ ] 종일 일정은 종료일을 포함하지 않는 날짜 범위로 표시되고 일반 일정은 현지 시작·종료 시각을 표시한다.
- [ ] `America/New_York` 그룹의 DST 공백·중복 현지 시각 입력이 명확한 오류로 거부된다.
- [ ] 같은 일정을 두 탭에서 변경하면 오래된 수정·삭제가 409로 차단된다.
- [ ] 다른 그룹 계정은 그룹 캘린더 조회와 일정 ID 직접 변경 모두 차단된다.

## 9. 대시보드

- [ ] 홈에서 오늘 마감·지연·진행 중·읽지 않은 알림 수가 실제 내 담당 업무와 일치한다.
- [ ] 지연 업무가 긴급·높음 우선순위보다 먼저 표시되고 종료 업무는 내 우선 업무에서 제외된다.
- [ ] 활성 참여 그룹별 담당·진행·완료·지연 수와 향후 7일 일정, 최근 알림이 표시된다.
- [ ] 그룹 대시보드 상태별 건수 합계와 전체 업무 수가 일치한다.
- [ ] 기간 완료율·기한 준수율·흐름 진행률을 문서 공식으로 직접 계산한 결과와 화면이 일치한다.
- [ ] 지표 분모가 0이면 `0%`가 아니라 `-`로 표시된다.
- [ ] 팀원별 담당·진행·완료·지연과 기한 준수율이 실제 담당 업무와 일치한다.
- [ ] `LEADER_ONLY` 그룹 대시보드는 MEMBER 화면에서 조회되지 않으며 직접 API도 403으로 차단된다.
- [ ] 기간 역전 또는 366일 초과 요청이 `DASHBOARD_RANGE_INVALID`로 거부된다.
- [ ] 위험 업무를 선택하면 해당 업무 상세로 이동한다.

## 10. PWA와 접근성

- [ ] `cd frontend && npm run build && npm run preview`로 실행했을 때 Manifest와 Service Worker가 등록된다.
- [ ] 지원 브라우저에서 설치 안내가 나타나고 설치 후 독립 창으로 시작된다.
- [ ] 새 빌드를 배포하고 새로고침하면 업데이트 안내가 나타나며 `업데이트` 후 최신 화면으로 전환된다.
- [ ] 오프라인 전환 시 안내가 표시되고 API 조회·변경이 캐시 응답으로 성공한 것처럼 보이지 않는다.
- [ ] 키보드만으로 본문 건너뛰기, 메뉴, 폼, 캘린더 이동, 알림 목록을 순서대로 조작할 수 있다.
- [ ] 모든 조작 요소에 포커스 표시가 보이고 200% 확대에서도 내용과 조작 요소가 겹치지 않는다.
- [ ] 운영체제의 동작 줄이기 설정에서 불필요한 전환 효과가 제거된다.

- [x] 모바일 360px 너비에서 필드·버튼·그룹·업무 카드가 잘리지 않는다.
