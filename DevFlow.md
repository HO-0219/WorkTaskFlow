# 개발 흐름(DevFlow)

이 문서는 프로젝트의 구현 순서, 현재 단계, 단계별 완료 조건을 관리한다.

- 제품 요구사항과 도메인 규칙: [`프로젝트_기초구조_설명.txt`](./프로젝트_기초구조_설명.txt)
- 로컬 실행과 환경 설정: [`README.md`](./README.md)
- 실제 작업과 검증 기록: [`devLog/`](./devLog/)
- 이 문서: 앞으로 무엇을 어떤 순서로 개발할지 관리

요구사항이 바뀌면 기초 구조 설명을 먼저 수정하고, 개발 순서가 바뀌면 이 문서를 수정한다. 실제로 수행한 작업은 추측하지 않고 날짜별 개발 로그에 남긴다.

## 상태 표시

- `완료`: 단계의 완료 게이트를 통과했다.
- `진행`: 현재 작업 중이다.
- `대기`: 선행 단계가 끝난 뒤 시작한다.
- `보류`: 범위 또는 정책 결정이 필요하다.

## 개발 운영 원칙

1. 한 단계의 완료 게이트를 통과한 뒤 다음 단계로 이동한다.
2. 기술 계층 전체를 먼저 만드는 대신 `요구사항 → 도메인/DB → API → 화면 → 테스트 → 문서`의 세로 기능 단위로 완성한다.
3. API, DB, 권한, 환경변수, 정책 변경은 같은 작업의 개발 로그에 기록한다.
4. 기능 작업은 `feature/{domain}-{feature}` 브랜치에서 진행한다.
5. 다른 도메인의 Entity나 Repository를 편의상 직접 사용하지 않는다. 필요한 계약은 application 서비스, DTO 또는 이벤트로 공개한다.
6. 백엔드 Entity를 프런트엔드 타입으로 그대로 복제하지 않는다.
7. 미완성 기능은 베타·발표 빌드에 노출하지 않는다.
8. 기능 완료 전 백엔드 테스트와 프런트엔드 빌드를 실행한다.
9. 비밀번호, 토큰, 인증번호, API Key, 개인정보 원문은 코드·Git·개발 로그에 남기지 않는다.
10. AI 리포트는 별도 담당 트랙으로 개발하며 핵심 업무 처리를 막지 않는다.
11. 기능 개발과 로컬 QA의 기준 DB는 로컬 MySQL 8.x로 한다. H2는 빠른 격리 테스트 보조 수단으로만 사용한다.
12. Docker, AWS, S3, CI/CD는 핵심 기능과 로컬 QA 완료 후 별도 인프라 단계에서 함께 구성한다.
13. MySQL 자동 테스트는 `teamProject_test` 전용 DB에서만 실행하고 Flyway 개발 DB `teamProject`에는 `create-drop`을 적용하지 않는다.

## AI 리포트 별도 병렬 트랙

AI 담당 트랙은 핵심 개발과 병렬로 진행하되, 확정 통계 계약이 준비되기 전에는 운영 데이터에 연결하지 않는다.

### AI-0. 계약과 지표 정의

- 리포트 지표와 위험 판단 규칙 합의
- 개인정보가 제거된 `MetricsSnapshot` 계약 정의
- Task, User 등 원본 Entity/Repository 직접 접근 금지

### AI-1. 테스트 데이터와 품질 기준

- 합성 데이터 기반 프롬프트와 응답 스키마 설계
- 프롬프트·모델 설정·출력 스키마 버전 관리
- 과장, 근거 없는 순위화, 수치 불일치 평가표 작성

### AI-2. 리포트 연결

- 서버가 확정한 metricsJson만 입력
- AI는 수치를 새로 계산하거나 변경하지 않음
- 결과 검증 실패 또는 AI 장애 시 기본 통계·PDF 제공

### AI-3. 베타 QA

- timeout, retry, rate limit, 비용 제한
- 빈 데이터, 장애, 잘못된 응답 테스트
- 개인정보가 외부 요청과 로그에 포함되지 않는지 검증

### AI 완료 게이트

- AI 기능을 꺼도 기본 통계와 PDF가 생성된다.
- 확정 수치와 AI 문장의 불일치를 탐지한다.
- 개인정보, 댓글 전문, 파일 내용이 전달되지 않는다.
- 사람을 근거 없는 단일 점수로 평가하지 않는다.

4. application 유스케이스와 권한 검증 구현
5. presentation API와 공통 오류 연결

8. 백엔드 테스트와 프런트 빌드
9. README, 기초 구조 문서, DevFlow, 개발 로그 갱신
10. 리뷰 후 병합하고 후속 작업 기록

## 공통 완료 정의(Definition of Done)

- 요구사항과 권한 규칙을 충족한다.
- Entity를 API 응답으로 직접 반환하지 않는다.
- 입력값과 서버 권한을 모두 검증한다.
- 정상, 실패, 401/403, 잘못된 상태, 동시 수정 테스트가 있다.
- API, DB, 환경변수, 정책 변경이 문서와 개발 로그에 기록되어 있다.
- `cd backend && ./mvnw test`가 통과한다.
- `cd frontend && npm run build`가 통과한다.
- PC Chrome과 모바일 크기에서 확인한다.
- 로딩, 빈 결과, 오류, 권한 없음 상태를 확인한다.
- timezone, 개인정보, 비밀키 노출 여부를 확인한다.
- 미완료 항목과 다음 작업이 개발 로그에 남아 있다.

## 개발 로그 작성 규칙

- 기본 파일명: `devLog/YYYY-MM-DD.md`
- 같은 날짜에 독립 작업이 여러 개면 `YYYY-MM-DD-{topic}.md`를 사용한다.
- 실제 수행한 작업과 검증 결과만 기록한다.
- 명령이 실패했으면 실패 원인과 해결 여부를 함께 적는다.
- API, DB, 환경변수, 권한, 호환성 영향을 명시한다.
- 비밀번호, 토큰, 인증번호, API Key, 개인정보 원문은 기록하지 않는다.
- 단계의 완료 게이트를 통과했을 때만 이 문서의 상태를 `완료`로 변경한다.

개발 로그의 기본 목차:

```text
# YYYY-MM-DD 개발 로그
## 작업 단계
## 오늘의 목표
## 완료한 작업
## 결정 사항
## 영향 범위
## 검증
## 이슈 및 주의사항
## 미완료 및 다음 작업
```
