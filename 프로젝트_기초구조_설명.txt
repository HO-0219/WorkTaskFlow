업무 프로세스 관리 PWA 프로젝트 기초 구조 및 개발 방향
========================================================

1. 프로젝트 개요
----------------

이 프로젝트는 개인과 팀의 업무를 등록하고, 배분하고, 진행 상태를 추적하며,
캘린더·대시보드·AI 리포트로 업무 수행 현황을 확인하는 업무 프로세스 관리 웹앱이다.

하나의 React 웹앱을 PC와 모바일 브라우저에서 반응형으로 제공하고,
Web App Manifest와 Service Worker를 적용하여 설치 가능한 PWA로 만든다.

핵심 콘셉트:

  "개인 ToDo와 그룹 업무 관리를 하나로 연결하는 협업형 업무 프로세스 관리 PWA"

주요 가치:

- 개인 업무와 여러 그룹의 업무를 한 계정에서 관리
- 팀원이 업무를 제안하고 팀장이 수락·배분·할당하는 명확한 프로세스
- 팀장이 업무를 제안하고 배분·할당하는 프로세스
- 업무, 일정, 댓글, 첨부파일, 체크리스트를 한곳에서 관리
- 개인/그룹 캘린더와 통합 대시보드 제공
- 주간·월간·연간 업무 데이터를 분석한 AI 리포트 제공
- 역할과 이벤트에 따른 앱 내 알림 및 이메일 리포트 전달 (추후 확장 SNS, SMS 알림)
- PC/모바일 반응형 웹과 설치 가능한 PWA 지원


2. 핵심 용어와 권한 기준
-------------------------

User(사용자):

- 이름, 닉네임, 이메일, 비밀번호, 전화번호 등의 계정 정보를 가진다.
- 한 사용자는 여러 그룹에 참여할 수 있다.
- 시스템 관리자 권한과 그룹 내 권한은 분리한다.

Group(그룹):

- PERSONAL: 가입할 때 자동 생성되는 1인 개인 그룹
- TEAM: 여러 사용자가 참여할 수 있는 협업 그룹
- 한 사용자에게 개인 그룹은 정확히 하나만 존재한다.
- 팀 그룹은 여러 개 생성하거나 가입할 수 있다.

GroupMember(그룹 멤버십):

- 사용자와 그룹의 관계를 나타낸다.
- 역할은 LEADER(팀장), MEMBER(팀원)로 구분한다.
- 권한은 사용자 계정 전체가 아니라 그룹별로 부여한다.
- 따라서 같은 사용자가 A그룹에서는 팀장, B그룹에서는 팀원일 수 있다.
- 개인 그룹의 사용자는 해당 그룹의 유일한 LEADER이다.

권한 요약:

- 팀원: 업무 등록, 자신에게 할당된 업무 조회·수행, 댓글·체크리스트 작성
- 팀장: 팀원 권한 + 업무 수락/반려, 담당자 분배·할당, 그룹 설정 및 멤버 관리
- 그룹 대시보드 공개 범위와 리포트 수신 설정은 팀장이 관리
- 모든 조회와 변경은 현재 사용자가 해당 그룹의 멤버인지 먼저 검증


3. 팀 인원과 개발 범위
-----------------------

전체 팀원은 5명이지만 주력 개발 인원은 2명으로 가정한다.
1차에서는 업무 등록부터 완료, 알림, 기본 통계까지 하나의 흐름을 완성하는 데 집중한다.

MVP(1차 필수 범위):

- 기존 회원가입, 로그인, 이메일 인증, 비밀번호 재설정 안정화
- 사용자 프로필: 이름, 닉네임, 이메일, 전화번호
- 개인 그룹 자동 생성
- 팀 그룹 생성, 수정, 멤버 초대/가입, 팀장/팀원 권한
- 업무 등록, 수락/반려, 할당, 시작, 보류, 중단, 완료
- 업무 우선순위, 마감일, 담당자, 체크리스트
- 업무 댓글과 멘션
- 앱 내 알림
- 개인/그룹 캘린더와 기본 일정 작성
- 개인 대시보드와 기본 그룹 대시보드
- PWA 설치와 반응형 레이아웃

2차 확장 범위:

- 그룹 자료/업무 첨부파일 업로드·다운로드와 외부 협업 링크 등록
- 알파·베타에서는 로컬 또는 서버 영구 볼륨을 사용하고, 발표 후 운영 준비 단계에서 S3로 전환
- 캘린더 일정 반복 규칙과 외부 캘린더 연동
- 그룹 업무/일정을 개인 캘린더로 가져오기
- 이메일 알림과 예약 발송
- 주간 PDF 리포트 자동 생성·전송
- 월간/연간 리포트 및 AI 서술형 분석
- 브라우저 Push 알림
- 실시간 알림(SSE 또는 WebSocket)
- 감사 로그, 고급 검색, 데이터 내보내기

현재 제외 범위:

- 메신저, 친구, 1:1 채팅, 그룹 채팅
- 게임봇과 미니게임
- 영상 회의와 음성 통화
- 급여, 근태 정산, 전자결재 전체 기능
- 네이티브 모바일 앱
- Kafka 등 현재 규모에 불필요한 대규모 메시징 인프라
- AI 모델 자체 학습


4. 현재 저장소 상태와 전환 원칙
----------------------------------

현재 저장소는 업무 관리 완성본이 아니라 인증 기능이 구현된 스타터 프로젝트이다.

현재 구현된 기능:

- 아이디/비밀번호 회원가입과 로그인
- 로그아웃
- 이메일 인증
- 아이디 찾기
- 비밀번호 재설정
- JWT Access Token
- HttpOnly Refresh Token
- Google/Kakao OAuth2 기본 구조
- 현재 로그인 사용자 조회
- 헬스 체크 API
- React 인증 화면

현재 User 엔티티에는 username, email, passwordHash, name, role 등이 있다.
닉네임과 전화번호, 프로필 상태 등은 업무 관리 요구사항에 맞게 추가해야 한다.

전환 시 유지하는 것:

- 인증, 이메일 인증, 토큰 재발급, OAuth2 기반 구조
- Spring Security 공통 설정과 공통 예외 처리
- React/Vite 기본 구조
- 로컬 MySQL 실행 환경(Docker Compose는 후속 인프라 단계)

전환 시 제거하거나 사용하지 않는 것:

- 메신저, 친구, 채팅방, 채팅 메시지 설계
- WebSocket/STOMP 채팅 구조
- 게임봇, 게임 세션, 이미지 필터링 중심 설계
- 채팅 내역 캐시와 CSV/TXT 대화 내보내기 계획

중요:

- 현재 User.role의 USER/ADMIN은 시스템 권한으로만 사용한다.
- 팀장/팀원은 User.role에 추가하지 않고 GroupMember.role로 관리한다.
- 애플리케이션 시작 클래스는 com.teamproject 루트에 두고 user, authentication, authorization, jwt 패키지를 모두 스캔한다.
- 인증 기능의 외부 API·DB·토큰 계약을 유지한 채 사용자, 인증, 인가, JWT 책임으로 패키지 분리를 완료했다.


5. 기술 스택
-------------

백엔드:

- Java 21
- Spring Boot 3
- Maven Wrapper
- Spring Security
- JWT Access Token + HttpOnly Refresh Token
- Spring Data JPA
- Bean Validation
- Spring Scheduling: 리포트 생성과 예약 발송
- Spring Mail: 이메일 알림과 리포트 전송
- MySQL 8.x
- PDF 라이브러리: OpenHTMLtoPDF 등 구현 시 선정
- 파일 저장소: 알파는 backend/uploads, 베타는 서버 영구 볼륨, 운영은 S3 호환 저장소

프런트엔드:

- React 18
- TypeScript
- Vite
- React Router
- 서버 상태 관리 도구는 도입 시 TanStack Query 우선 검토
- 캘린더 UI 라이브러리는 요구사항 확정 후 선정
- PWA Manifest + Service Worker
- IndexedDB: 오프라인 임시 작업과 최근 조회 데이터 캐시 후보

인프라 및 외부 서비스:

- MySQL: 사용자, 그룹, 업무, 일정, 댓글, 알림, 리포트 원본 데이터
- SMTP 또는 이메일 서비스: 알림과 PDF 리포트 발송
- 서버 영구 볼륨: 베타 테스트의 그룹 자료와 첨부파일 보관
- S3 호환 저장소: 발표 후 출시 후보 환경의 첨부파일 보관
- AWS EC2 또는 동등 환경: Spring Boot 배포 후보
- AI API: 리포트의 서술형 요약을 만들 때 선택적으로 연동

초기에는 단일 Spring Boot 서버와 MySQL로 시작한다.
Redis, 메시지 큐, 검색 엔진은 실제 성능 또는 운영 요구가 확인될 때 추가한다.

7. 사용자와 그룹 도메인
------------------------

User:

- userId
- username: 로그인 아이디
- name: 실명 또는 표시 이름
- nickname: 서비스 내 닉네임
- email
- passwordHash
- phoneNumber
- profileImageUrl (선택)
- systemRole: USER, ADMIN
- status: ACTIVE, SUSPENDED, WITHDRAWN
- emailVerified
- createdAt, updatedAt, lastLoginAt

비밀번호 원문은 저장하지 않고 안전한 해시만 저장한다.
이메일과 로그인 아이디의 중복 정책을 명확히 하고 DB Unique 제약조건을 둔다.

Group:

- groupId
- type: PERSONAL, TEAM
- name
- description (선택)
- dashboardVisibility: LEADER_ONLY, MEMBERS
- reportEnabled
- weeklyReportDay
- weeklyReportTime
- timezone
- createdBy
- createdAt, updatedAt

GroupMember:

- groupMemberId
- groupId
- userId
- role: LEADER, MEMBER
- status: INVITED, ACTIVE, LEFT, REMOVED
- joinedAt

그룹 규칙:

- 회원가입 완료 시 개인 그룹과 개인 그룹 멤버십을 한 트랜잭션으로 생성한다.
- 개인 그룹은 초대, 탈퇴, 삭제, 팀장 위임을 허용하지 않는다.
- 팀 그룹에는 항상 최소 한 명의 LEADER가 있어야 한다.
- 마지막 팀장은 탈퇴할 수 없으며 먼저 다른 멤버에게 팀장을 위임해야 한다.
- 팀 그룹 이름은 그룹 안에서 표시용이며 전체 서비스에서 중복될 수 있다.
- 그룹 초대는 이메일 또는 사용자 검색 방식으로 확장할 수 있다.


8. 업무(Task) 도메인
---------------------

Task:

- taskId
- groupId
- title
- description
- requesterId: 업무 등록자
- approverId: 수락 또는 반려한 팀장
- assigneeId: 현재 담당자, 미할당 가능
- priority: LOW, NORMAL, HIGH, URGENT
- status
- startAt (선택)
- dueAt (선택)
- completedAt (선택)
- holdReason (선택)
- stopReason (선택)
- createdAt, updatedAt

권장 업무 상태:

- REQUESTED: 팀원이 등록하여 팀장 검토를 기다리는 상태
- TODO: 팀장이 수락했지만 아직 시작하지 않은 상태
- IN_PROGRESS: 담당자가 수행 중인 상태
- ON_HOLD: 사유와 함께 일시 보류한 상태
- COMPLETED: 완료한 상태
- REJECTED: 팀장이 반려한 상태
- CANCELLED: 권한 있는 사용자가 취소한 상태

DELAYED(지연)는 별도 저장 상태로 두기보다 다음 조건으로 계산한다.

  dueAt < 현재 시각 AND status NOT IN (COMPLETED, REJECTED, CANCELLED)

이렇게 하면 마감일이 변경되거나 업무가 완료될 때 지연 여부가 자동으로 정확해진다.

기본 업무 흐름:

  팀원이 업무 등록
      -> REQUESTED
      -> 팀장 알림
      -> 팀장 수락 및 담당자 할당
      -> TODO
      -> 담당자 시작
      -> IN_PROGRESS
      -> 완료
      -> COMPLETED
      -> 팀장 완료 알림

예외 흐름:

- 팀장 반려: REQUESTED -> REJECTED, 반려 사유 필수
- 업무 보류: IN_PROGRESS -> ON_HOLD, 보류 사유 필수
- 업무 재개: ON_HOLD -> IN_PROGRESS
- 업무 중단: TODO/IN_PROGRESS/ON_HOLD -> CANCELLED, 중단 사유 필수
- 담당자 변경: 팀장만 가능하며 이전/새 담당자 모두에게 알림
- 개인 그룹의 업무는 승인 없이 생성 즉시 TODO로 전환 가능

권한 규칙:

- 팀원은 자신이 등록한 REQUESTED 업무를 팀장이 처리하기 전까지 수정/취소할 수 있다.
- 팀장은 그룹의 요청 업무를 수락·반려하고 담당자를 할당할 수 있다.
- 담당자는 자신에게 할당된 업무를 시작·보류·재개·완료할 수 있다.
- 팀장은 예외 상황에서 상태를 변경할 수 있지만 사유와 변경 이력을 남긴다.
- 모든 업무 변경은 그룹 멤버십과 역할을 서버에서 검증한다.

TaskStatusHistory:

- historyId
- taskId
- fromStatus, toStatus
- changedBy
- reason (선택 또는 상태에 따라 필수)
- changedAt

상태 이력은 알림, 리포트, 감사 추적, 수행 시간 계산의 근거가 되므로 삭제하지 않는다.


9. 체크리스트, 그룹 자료, 첨부파일, 댓글
-------------------------------------------

TaskChecklistItem:

- checklistItemId
- taskId
- content
- completed
- completedBy, completedAt
- sortOrder
- createdAt, updatedAt

체크리스트 진행률:

  완료 항목 수 / 전체 항목 수 * 100

체크리스트가 없을 때는 0%가 아니라 "체크리스트 없음"으로 표시한다.
업무 완료 시 미완료 체크리스트 허용 여부는 그룹 정책으로 확장할 수 있다.

Resource:

- resourceId
- groupId: 모든 자료가 속한 그룹
- taskId (선택): 특정 업무에 첨부한 자료일 때 사용
- type: FILE, LINK
- title
- originalFileName (FILE일 때)
- storageKey (FILE일 때)
- storageProvider: LOCAL, S3 (FILE일 때)
- externalUrl (LINK일 때)
- linkProvider: GITHUB, JIRA, NOTION, GOOGLE_DRIVE, OTHER (LINK일 때)
- contentType, fileSize, checksum (FILE일 때)
- uploadedBy
- createdAt

그룹 제안서·기획서처럼 특정 업무에 속하지 않는 자료는 taskId를 비워 그룹 자료로 관리한다.
업무와 연결된 자료는 taskId를 저장하여 업무 상세와 그룹 자료 목록에서 함께 조회할 수 있게 한다.
GitHub, Jira, Notion, Google Drive 등은 파일을 복제하지 않고 LINK 자료로 등록한다.

저장 경로 규칙:

  backend/uploads/
  └── groups/
      └── group-{groupId}/
          └── {yyyy}/
              └── {MM}/
                  └── {yyyyMMdd'T'HHmmss'Z'}_member-{memberId}_{uuid}.{ext}

예시:

  backend/uploads/groups/group-15/2026/07/20260721T063045Z_member-42_a1b2c3d4.pdf

로컬에서는 위 구조를 실제 디렉터리로 만들고, S3에서는 backend/uploads를 제외한 groups/... 부분을 object key prefix로 사용한다.

사용자가 그룹 디렉터리나 저장 경로를 직접 선택하지 않는다.
서버가 활성 그룹 멤버십과 자료 등록 권한을 검증한 후 그룹 디렉터리가 없으면 생성하고 있으면 재사용한다.
그룹명은 중복될 수 있고 변경될 수 있으며 특수문자가 포함될 수 있으므로 실제 디렉터리 식별자로 사용하지 않는다.
화면에는 groupId로 조회한 현재 그룹명을 표시하고 실제 저장 경로는 group-{groupId}로 고정한다.
올린 멤버의 이름이나 닉네임도 변경될 수 있으므로 파일명에는 해당 그룹의 groupMemberId를 memberId로 사용하고, 화면에서 DB의 현재 멤버 정보를 표시한다.
동일한 초에 같은 멤버가 여러 파일을 올리는 경우를 대비해 uuid를 붙인다.
시각은 서버의 UTC 기준으로 생성하고 Z로 표시한다.
원본 파일명은 물리 경로에 사용하지 않고 DB 메타데이터에만 저장하여 다운로드 시 표시한다.
확장자는 원본 파일명을 그대로 신뢰하지 않고, 서버가 검증한 허용 파일 형식에서 결정한다.

저장소 전환 원칙:

- 알파(로컬): 프로젝트의 backend/uploads에 저장한다.
- 베타(클라우드 서버): 재배포와 컨테이너 재시작 후에도 유지되는 서버 영구 볼륨에 저장한다.
- 발표 후 출시 후보: 운영과 같은 조건의 스테이징에서 S3로 전환하고 이전·권한·장애 테스트를 통과한다.
- 실제 운영: S3 비공개 버킷을 사용하고 권한 검증 후 만료형 URL로 업로드·다운로드한다.

애플리케이션은 FileStorage 포트만 의존하고 LocalFileStorage와 S3FileStorage가 같은 storageKey 규칙을 사용한다.
DB에는 backend/uploads와 같은 절대 경로나 S3 공개 URL을 저장하지 않고 상대 storageKey만 저장한다.
저장소 루트는 app.storage.local.root 같은 환경 설정으로 바꾸며 빌드 산출물과 컨테이너 내부 임시 경로에 묶지 않는다.
업로드 디렉터리의 실제 파일은 Git에 추적하지 않는다.
파일 크기, 허용 확장자, MIME 타입, 실제 파일 형식, 저장 용량, 다운로드 권한을 서버에서 검증한다.

Comment:

- commentId
- taskId
- authorId
- content
- createdAt
- updatedAt (수정하지 않았다면 null 가능)
- deletedAt (소프트 삭제 시 사용)

CommentMention:

- mentionId

- mentionedUserId

댓글 규칙:

- 해당 업무가 속한 그룹의 활성 멤버만 댓글을 조회하고 작성할 수 있다.
- 댓글 수정/삭제는 작성자 또는 정책상 허용된 팀장만 할 수 있다.
- 멘션 대상은 해당 그룹의 활성 멤버로 제한한다.
- 일반 댓글은 업무 담당자에게 알림을 보낸다.
- 멘션된 사용자는 담당자 여부와 관계없이 멘션 알림을 받는다.
- 동일 이벤트에서 같은 사용자에게 알림이 중복 생성되지 않게 한다.


10. 캘린더 도메인
------------------

CalendarEvent:

- eventId
- groupId
- createdBy
- type: SCHEDULE, MEETING, VACATION, DEADLINE, TODO
- title
- description
- startAt, endAt
- allDay
- location (선택)
- sourceTaskId (업무 마감 일정이면 사용)
- createdAt, updatedAt

캘린더 구분:

- 개인 캘린더: PERSONAL 그룹에 속한 일정과 개인 업무 마감일
- 그룹 캘린더: 해당 TEAM 그룹의 일정과 그룹 업무 마감일
- 캘린더 화면에서는 그룹별 색상과 필터를 제공한다.

업무 마감일 연동:

- dueAt이 있는 업무는 캘린더에서 DEADLINE 항목으로 조회한다.
- 업무 마감일을 별도 일정으로 복제하지 않고 sourceTaskId로 연결하거나 조회 시 합성한다.
- 업무 마감일 변경 시 캘린더에도 즉시 반영되어야 한다.

그룹 항목을 개인 캘린더로 가져오기:

- 사용자가 그룹 업무 또는 일정을 개인 캘린더에 표시하도록 선택할 수 있다.
- 원본을 복사해 독립 데이터로 만들기보다 CalendarSubscription 또는 PersonalCalendarLink로 연결한다.
- 원본 제목·시간이 바뀌면 개인 캘린더에도 반영된다.
- 사용자는 개인 캘린더에서 연결을 해제할 수 있지만 그룹 원본은 삭제할 수 없다.
- 권한을 잃거나 그룹에서 탈퇴하면 연결 항목을 더 이상 조회할 수 없다.

시간 처리:

- 서버와 DB에는 가능한 한 UTC로 저장한다.
- 화면 표시와 예약 발송은 사용자 또는 그룹 timezone을 사용한다.
- 시작 시각은 종료 시각보다 늦을 수 없도록 검증한다.


11. 대시보드
-------------

개인 대시보드:

- 개인 그룹 ToDo 요약
- 오늘/이번 주 마감 업무
- 지연 업무와 우선순위 높은 업무
- 참여 중인 모든 그룹의 내 담당 업무
- 그룹별 완료/진행/보류 업무 수
- 최근 알림과 다가오는 일정

개인 대시보드는 사용자가 여러 그룹에서 진행한 자신의 업무를 한 번에 보는 통합 화면이다.

그룹 대시보드:

- 전체 업무량
- 상태별 업무 수: 요청, 할 일, 진행, 보류, 완료, 지연
- 전체 진행률과 기간별 완료율
- 팀원별 할당 업무 수와 상태
- 팀원별 기한 내 완료율
- 위험 업무/프로젝트 요약
- 기간별 업무 생성·완료 추이

공개 범위:

- LEADER_ONLY: 팀장만 그룹 대시보드 조회 가능
- MEMBERS: 해당 그룹의 모든 활성 멤버가 조회 가능
- 공개/비공개 설정은 팀장만 변경 가능
- MEMBERS여도 이메일, 전화번호 등 불필요한 개인정보는 노출하지 않는다.

지표 정의 예시:

- 완료율 = 기간 내 완료 업무 수 / 기간 내 처리 대상 업무 수
- 기한 준수율 = 마감일까지 완료한 업무 수 / 마감일이 있는 완료 업무 수
- 진행률 = 가중 상태 점수 또는 체크리스트 기준으로 계산
- 처리량 = 기간 내 완료된 업무 수

"수행능력"과 "능률"은 사람을 단순 평가하는 점수로 고정하지 않는다.
업무 난이도와 외부 요인을 고려하지 않은 순위화는 오해를 만들 수 있으므로,
초기에는 처리량, 완료율, 기한 준수율, 평균 소요 시간 등 설명 가능한 지표를 나란히 제공한다.


12. AI 리포트
--------------

AI 리포트는 계산 가능한 통계와 AI가 작성하는 서술형 요약을 분리한다.

1) 서버가 원본 데이터로 수치와 지표를 확정한다.
2) 확정된 집계 데이터만 AI에 전달하여 요약, 위험 신호, 개선 제안을 생성한다.
3) AI는 업무 수, 완료율 같은 원본 수치를 임의로 계산하거나 변경하지 않는다.
4) AI 호출에 불필요한 개인정보, 댓글 전문, 첨부파일 내용을 보내지 않는다.
5) AI 호출 실패 시에도 통계 리포트와 PDF는 생성할 수 있어야 한다.

주간 리포트:

- 한 주의 총 업무, 완료, 진행 중, 보류, 지연 건수
- 위험 업무/프로젝트 여부와 판단 근거
- 팀장을 포함한 팀원별 담당 업무와 수행 현황
- 월~금 5일간의 생성·완료 추이와 업무 집중도
- 다음 주 우선 확인 항목과 개선 제안

월간 리포트:

- 월간 총 업무, 완료, 진행 중, 보류, 지연 건수
- 위험 업무/프로젝트 여부와 변화
- 팀장을 포함한 팀원별 담당 업무와 수행 현황
- 월초·월중·월말의 처리량과 기한 준수 추이
- 그룹 전체의 업무 수행 경향과 병목 분석

연간 리포트:

- 연간 업무 총량과 상태별/월별 추이
- 그룹의 전반적인 수행 업무와 완료 경향
- 기한 준수율, 평균 처리 시간, 보류/지연 변화
- 주요 성과, 반복 병목, 다음 해 개선 제안

위험 판단 예시:

- 지연 업무 비율이 그룹 기준치를 초과
- 마감 임박 업무 중 미시작 업무가 다수 존재
- 특정 담당자에게 URGENT/HIGH 업무가 과도하게 집중
- 장기간 ON_HOLD 상태인 업무가 존재
- 최근 기간의 완료율이 이전 기간보다 크게 하락

판단 기준과 기준값은 코드에 흩어놓지 않고 ReportPolicy 또는 그룹 설정으로 관리한다.
리포트에는 "왜 위험으로 판단했는지" 근거를 함께 표시한다.


13. 리포트 활성화와 전달 규칙
--------------------------------

주간 리포트:

- 개인 그룹을 제외한 TEAM 그룹이 대상이다.
- 팀장이 설정한 요일과 시각에 그룹 timezone 기준으로 생성한다.
- PDF로 변환한 뒤 해당 그룹의 활성 팀장에게 이메일로 전송한다.
- 같은 그룹/기간의 리포트가 중복 발송되지 않도록 발송 이력을 저장한다.

월간 리포트:

- 그룹 생성일 또는 유효 업무 데이터 시작일부터 14일 이상일 때 다운로드 가능하다.
- 14일 미만이면 다운로드 버튼은 화면에 보이지만 disabled 상태로 표시한다.
- 비활성 사유와 활성 예정일을 버튼 주변에 안내한다.

연간 리포트:

- 그룹 생성일 또는 유효 업무 데이터 시작일부터 3개월 이상일 때 제공한다.
- 3개월 미만이면 버튼이나 메뉴 자체를 표시하지 않는다.

Report:

- reportId
- groupId
- type: WEEKLY, MONTHLY, YEARLY
- periodStart, periodEnd
- status: PENDING, GENERATING, COMPLETED, FAILED
- metricsJson: 생성 당시 확정 지표 스냅샷
- aiSummary (선택)
- fileUrl 또는 objectKey (선택)
- generatedAt

ReportDelivery:

- deliveryId
- reportId
- recipientUserId
- channel: EMAIL
- status: PENDING, SENT, FAILED
- attemptedAt, sentAt
- failureReason (선택)

ReportScheduler는 리포트 생성과 이메일 발송을 분리한다.
실패 시 재시도할 수 있어야 하며, 재시도해도 동일 리포트가 중복 생성·전송되지 않게 한다.


14. 알림 규칙
-------------

필수 알림 이벤트:

- 팀원이 업무 등록 -> 해당 그룹 팀장에게 알림
- 팀장이 업무 수락/반려 -> 업무 등록자에게 알림
- 팀장이 업무 할당 -> 담당 팀원에게 알림
- 담당자가 업무 시작 -> 팀장에게 진행 시작 알림
- 담당자가 업무 완료 -> 팀장에게 완료 알림
- 담당자가 업무 보류/중단 -> 사유와 함께 팀장에게 알림
- 담당자 변경 -> 이전 담당자와 새 담당자에게 알림
- 댓글 작성 -> 업무 담당자에게 알림
- 댓글 멘션 -> 멘션된 사용자에게 알림
- 마감 임박/지연 -> 담당자와 정책상 필요한 팀장에게 알림

Notification:

- notificationId
- recipientId
- groupId (선택)
- type
- title
- content
- targetType, targetId: 클릭 시 이동할 대상
- readAt
- createdAt

알림 원칙:

- 업무 상태 변경과 알림 이벤트 기록은 가능한 한 같은 트랜잭션 경계에서 처리한다.
- 이메일 발송 실패가 업무 처리 자체를 롤백시키지 않게 한다.
- 본인이 발생시킨 이벤트를 본인에게 다시 알릴지는 이벤트별로 정한다.
- 동일 수신자/이벤트에 대한 중복 알림을 방지한다.
- 알림 목록은 최신순 커서 페이지네이션을 사용한다.
- 읽음 처리와 전체 읽음 기능을 제공한다.

1차에서는 앱 내 알림을 폴링 방식으로 구현할 수 있다.
실시간성이 필요해지면 SSE를 우선 검토하고, 양방향 통신 요구가 생길 때 WebSocket을 검토한다.

16. PWA와 오프라인 원칙
------------------------

Chrome 웹과 PWA는 별도 프런트엔드가 아니다.
하나의 React 앱에 Web App Manifest와 Service Worker를 추가한다.

캐시 가능:

- HTML, CSS, JavaScript, 폰트, 아이콘 등 정적 자원
- 공개 설정과 변경 빈도가 낮은 공통 데이터
- 사용자별로 분리된 최근 조회 목록의 제한적 로컬 캐시

무조건 캐시하지 않는 것:

- 로그인/토큰 재발급 응답
- 다른 사용자의 개인정보
- 권한 확인 없이 재사용될 수 있는 그룹 데이터
- PDF 리포트와 첨부파일 원본

오프라인 원칙:

- 네트워크가 없음을 명확히 표시한다.
- 캐시된 최근 업무/일정은 읽기 전용으로 보여줄 수 있다.
- 오프라인 수정 기능은 2차 범위로 두고, 도입 시 충돌 정책을 먼저 정의한다.
- 로그아웃 시 사용자별 IndexedDB와 민감한 캐시를 정리한다.
- 새 버전이 준비되면 사용자 작업을 방해하지 않는 업데이트 안내를 제공한다.

19. 백엔드 개발 원칙
---------------------

백엔드는 기술 종류보다 도메인 단위로 나눈다.

권장 도메인 내부 구조:

com/teamproject/task/
├── domain/             Entity, Value Object, Repository 인터페이스
├── application/        유스케이스 서비스, Command, Query
├── infrastructure/     JPA와 외부 연동 구현
└── presentation/       REST Controller, Request/Response DTO

의존성 방향:

  presentation -> application -> domain
  infrastructure -------------> domain

사용자·인증 기반 패키지 책임:

- user는 User 엔티티와 사용자 저장 계약의 유일한 소유자다.
- authentication은 회원가입, 로그인, 이메일 인증, 계정 복구, Refresh Token, OAuth 흐름을 조정한다.
- authorization은 공개/보호 API 규칙과 향후 그룹 역할·자원 소유권 인가 정책을 관리한다.
- jwt는 Access Token의 생성·검증과 Bearer 인증 필터만 담당한다.
- Refresh Token은 JWT가 아닌 난수 기반 세션 토큰이므로 authentication/domain/token에 둔다.
- 현재 인증 API 경로, JSON 필드, DB 테이블, JWT Claim, Refresh Cookie 계약은 패키지 이동 전후에도 유지한다.

필수 원칙:

- domain은 Controller, HTTP, 이메일 API를 알지 못한다.
- Controller에 비즈니스 로직을 작성하지 않는다.
- presentation에서 Repository를 직접 호출하지 않는다.
- 다른 도메인의 내부 Entity/Repository를 무분별하게 직접 사용하지 않는다.
- 트랜잭션 범위는 application 서비스에서 관리한다.
- 권한 검사는 화면 숨김에 의존하지 않고 API에서 수행한다.
- 날짜 계산에는 Clock을 주입하여 테스트 가능하게 만든다.
- 업무 상태 전이와 리포트 집계 규칙을 테스트로 고정한다.

로그인 사용자:

JWT 인증이 완료되면 Spring Security Authentication principal에 Long userId가 들어간다.
도메인 서비스에는 Authentication 전체가 아니라 userId를 전달한다.

도메인 이벤트:

- TaskRequested
- TaskAssigned
- TaskStarted
- TaskHeld
- TaskCompleted
- TaskCancelled
- CommentCreated

초기에는 Spring의 트랜잭션 이벤트 또는 Outbox 패턴 후보로 알림과 리포트 연계를 분리한다.
단, MVP 단계에서 구조를 과도하게 복잡하게 만들지 않고 실패 복구가 필요한 이메일부터 분리한다.


20. 프런트엔드 개발 원칙
-------------------------

- 페이지 컴포넌트에 API 호출과 복잡한 상태 로직을 모두 작성하지 않는다.
- 기능별 features 폴더로 나눈다.
- API 요청/응답 타입과 화면 전용 타입을 구분한다.
- 백엔드 Entity를 프런트 타입으로 그대로 복사하지 않는다.
- 서버 상태와 입력 폼 상태를 구분한다.
- 권한에 따라 버튼을 숨기거나 비활성화하되 서버 검증을 전제로 한다.
- URL에 groupId, taskId, 날짜 필터를 반영하여 새로고침과 링크 공유를 지원한다.
- 로딩, 빈 결과, 권한 없음, 네트워크 오류 상태를 각각 설계한다.
- 날짜와 시간은 그룹/사용자 timezone 기준을 명확히 표시한다.
- PWA 설치 여부와 관계없이 핵심 기능은 동일하게 제공한다.


21. 보안과 개인정보 원칙
------------------------

- 비밀번호는 BCrypt 또는 승인된 PasswordEncoder로 해시한다.
- Access Token은 짧게 유지하고 Refresh Token은 HttpOnly/Secure Cookie로 관리한다.
- 모든 그룹 자원 조회에서 활성 멤버십을 검증한다.
- 팀장 전용 기능은 GroupMember.role을 검증한다.
- 첨부파일 다운로드에도 그룹/업무 접근 권한을 검증한다.
- 이메일, 전화번호는 필요한 화면과 권한에서만 노출한다.
- 로그에 비밀번호, 토큰, 인증번호, 이메일 전문 등 민감정보를 남기지 않는다.
- JWT Secret, SMTP 비밀번호, AWS Key, AI API Key를 Git에 추가하지 않는다.
- .env는 Git 추적에서 제외한다.
- 파일 이름을 신뢰하지 않고 크기, MIME 타입, 실제 포맷을 검증한다.
- 원본 파일명과 클라이언트가 전달한 경로를 저장 경로 생성에 사용하지 않고 path traversal과 symbolic link 우회를 차단한다.
- 로컬·서버 저장소의 실제 경로를 API 응답이나 로그에 노출하지 않는다.
- 댓글과 업무 설명은 출력 시 XSS에 안전하게 처리한다.
- 운영 환경에서는 HTTPS, Secure Cookie, CORS 허용 출처 제한을 적용한다.
- 리포트 PDF URL은 공개 URL 대신 권한 검사 또는 만료형 서명 URL을 사용한다.
- AI 리포트 연동 전 개인정보 처리 범위와 데이터 보존 정책을 확인한다.

- 업무 관리 서비스 벤치마킹
- 대시보드 정보 구조와 사용자 흐름
- 반응형/PWA 사용성 테스트

팀원 D(정책/테스트):

- 팀장/팀원 권한 시나리오
- 업무 상태 전이와 예외 시나리오
- 리포트 활성 조건과 알림 수신 조건 검증

팀원 E(문서/발표):

- 요구사항, ERD, API 명세 정리
- 테스트 결과와 오류 재현 기록
- 발표 자료와 시연 시나리오


23. 권장 개발 순서
-------------------

아래는 전체 순서의 요약이다. 현재 진행 상태, 세부 체크리스트, 단계별 완료 게이트는 루트의 DevFlow.md를 기준으로 관리한다.
실제로 수행한 작업과 검증 결과는 devLog/YYYY-MM-DD.md에 남긴다.

1) 현재 인증 테스트와 로컬 실행 상태 확인
2) User에 nickname, phoneNumber 등 프로필 필드 추가
3) Group, GroupMember와 개인 그룹 자동 생성
4) 그룹 생성, 초대, 가입, 역할/권한 API
5) Task와 상태 전이, 이력, 담당자 할당
6) 업무 목록/상세/등록과 반응형 기본 레이아웃
7) 체크리스트와 댓글/멘션
8) 앱 내 알림과 읽음 처리
9) 개인/그룹 캘린더와 업무 마감일 연동
10) 개인 대시보드와 그룹 대시보드 기본 지표
11) PWA Manifest, Service Worker, 설치/업데이트 테스트
12) 그룹 자료/업무 첨부와 LocalFileStorage 연동
13) 주간 리포트 집계, PDF 생성, 예약 이메일 전송
14) 월간/연간 활성 조건과 리포트 다운로드
15) AI 서술형 요약과 위험 분석 연동
16) 통합 테스트, 권한 테스트, 성능 점검, 베타 배포
17) 발표 후 S3FileStorage 구현, 기존 파일 이전, 출시 후보 테스트

먼저 1~11을 MVP로 완성하고, 리포트 통계의 원본 데이터가 충분히 쌓인 뒤 AI 연동을 진행한다.


24. 핵심 테스트 시나리오
-------------------------

사용자/그룹:

- 회원가입 시 개인 그룹이 정확히 한 번 생성된다.
- 한 사용자가 여러 팀 그룹에서 서로 다른 역할을 가질 수 있다.
- 일반 팀원이 팀장 전용 API를 호출하면 403을 반환한다.
- 마지막 팀장은 위임 없이 그룹을 탈퇴할 수 없다.
- 탈퇴한 멤버는 이전 그룹의 업무와 파일을 조회할 수 없다.

업무:

- 팀원 등록 업무가 REQUESTED가 되고 팀장 알림이 생성된다.
- 팀장 수락/할당 후 담당자만 업무를 시작할 수 있다.
- 허용되지 않은 상태 전이는 거부된다.
- 보류/중단/반려 시 사유가 없으면 실패한다.
- 마감 전 미완료 업무는 지연이 아니고, 마감 후에는 지연으로 계산된다.
- 동시 수정 시 한쪽 변경을 잃지 않는다.

댓글/알림:

- 그룹 외 사용자는 댓글을 읽거나 쓸 수 없다.
- 담당자와 멘션 대상이 같아도 알림은 한 번만 생성된다.
- 알림 읽음 처리 후 목록 상태가 일관된다.
- 이메일 실패가 업무 완료 처리를 취소하지 않는다.

그룹 자료/첨부파일:

- 그룹 외 사용자는 자료 목록·다운로드·삭제 API에 접근할 수 없다.
- 그룹명이 변경되어도 groupId 기반 storageKey로 기존 파일을 조회할 수 있다.
- 동일한 초에 같은 멤버가 여러 파일을 업로드해도 storageKey가 충돌하지 않는다.
- 허용되지 않은 확장자, 위조된 MIME 타입, 용량 초과, 경로 조작 시도를 거부한다.
- 서버 재시작·재배포 후에도 베타 영구 볼륨의 파일이 유지된다.
- DB 저장 실패, 디스크 부족, 파일 삭제 실패 시 메타데이터와 실제 파일의 일관성을 복구한다.
- S3 전환 시 기존 storageKey가 유지되고 체크섬과 파일 수를 비교한다.

캘린더/대시보드/리포트:

- 업무 마감일 변경이 캘린더에 반영된다.
- 개인 캘린더 연결 해제가 그룹 원본 일정을 삭제하지 않는다.
- 비공개 그룹 대시보드는 팀원에게 403을 반환한다.
- 월간 리포트는 14일 미만일 때 버튼만 비활성화된다.
- 연간 리포트는 3개월 미만일 때 UI에 표시되지 않고 API도 생성 요청을 거부한다.
- 예약 작업 재실행 시 같은 기간의 PDF와 이메일이 중복되지 않는다.


25. Git과 공통 파일 규칙
-------------------------

- main/develop에서 직접 기능 개발하지 않는다.
- feature/{domain}-{feature} 형식의 브랜치를 권장한다.
- 한 커밋에는 하나의 변경 목적을 유지한다.
- PR에 API, DB, 환경변수, 테스트 변경 내용을 기록한다.
- DB 스키마 변경에는 마이그레이션 도구 도입을 검토한다.
- backend/uploads의 실제 업로드 파일은 Git에 추적하지 않고 디렉터리 유지용 .gitignore만 추적한다.
- 다음 파일과 정책은 팀 전체에 영향을 주므로 수정 내용을 공유한다.
  - SecurityConfig와 JWT/Refresh Token
  - User, GroupMember 권한 모델
  - 업무 상태와 상태 전이 규칙
  - application.properties
  - pom.xml, package.json
  - docker-compose.yml
  - 공통 API 오류 형식
  - 자료 storageKey 규칙과 FileStorage 계약
  - 리포트 지표 공식과 활성화 기준


26. 작업 완료 기준
-------------------

- 요구사항과 권한 규칙을 충족한다.
- 입력값 검증과 그룹 멤버십 검증이 적용된다.
- 허용된 업무 상태 전이만 가능하다.
- 중요한 상태 변경과 사유가 이력으로 남는다.
- 실패 시 공통 API 오류 형식을 사용한다.
- 정상/권한 없음/잘못된 상태/동시 수정 테스트가 있다.
- ./mvnw test가 통과한다.
- npm run build가 통과한다.
- PC Chrome과 모바일 크기에서 확인한다.
- PWA 설치, 새 버전 업데이트, 오프라인 안내를 확인한다.
- 날짜/timezone 경계와 리포트 기간 계산을 테스트한다.
- 환경변수와 비밀키를 코드/Git에 추가하지 않는다.
- API/DB/환경변수/정책 변경 내용을 문서화한다.


27. 로컬 실행
-------------

필요 환경:

- Java 21
- Node.js 20 이상
- 로컬 MySQL 8.x

Docker, AWS, S3, CI/CD는 핵심 기능과 로컬 QA 완료 후 별도 인프라 단계에서 구성한다.

백엔드:

  cd project/backend
  ./mvnw spring-boot:run

프런트엔드:

  cd project/frontend
  npm install
  npm run dev

기본 주소:

- 프런트엔드: http://localhost:5174
- 백엔드: http://localhost:8081
- 상태 확인: http://localhost:8081/api/v1/health
- MySQL: localhost:3306/teamProject

로컬 DB 기본 설정:

- database: teamProject
- username: root
- password: test

로컬 비밀번호는 개발 전용이며 운영 환경에서 그대로 사용하지 않는다.
이메일 발송이 꺼져 있으면 인증번호와 비밀번호 재설정 링크는 백엔드 [LOCAL MAIL] 로그에서 확인한다.

로컬 업로드:

- 기본 저장소 루트: project/backend/uploads
- 환경 설정 예시: app.storage.type=local, app.storage.local.root=./uploads
- 백엔드를 backend 디렉터리에서 실행하면 ./uploads가 backend/uploads를 가리킨다.
- 실제 업로드 파일은 Git에 추적되지 않으며, 테스트 중 필요한 샘플 파일은 별도 test resources에 둔다.


28. 최종 원칙
-------------

- 개인 그룹은 가입 시 자동 생성되는 한 사용자의 기본 업무 공간이다.
- 팀장/팀원은 전역 권한이 아니라 그룹별 멤버십 권한이다.
- 업무 상태 전이와 변경 이력은 알림과 리포트의 신뢰 가능한 원본이다.
- 지연 여부는 저장값보다 마감일과 현재 상태로 계산한다.
- 업무 마감 일정은 중복 복사하지 않고 업무와 캘린더를 연결한다.
- 대시보드 지표는 정의와 계산 근거를 공개한다.
- AI는 확정된 통계를 설명하며 원본 수치를 임의로 만들지 않는다.
- 이메일/PDF/AI 실패가 핵심 업무 처리를 막지 않도록 분리한다.
- Chrome 웹과 PWA는 하나의 React 코드베이스로 구현한다.
- MVP 흐름을 완성한 뒤 첨부파일, 고급 리포트, 실시간 알림을 확장한다.
- 알파·베타의 파일은 로컬/서버 영구 볼륨에 저장하고 발표 후 출시 후보 QA를 거쳐 S3로 전환한다.
- 로컬과 S3는 같은 storageKey를 사용하며 저장 구현은 FileStorage 포트 뒤로 격리한다.
- 현재 규모에 필요한 설계부터 도입하고 복잡한 인프라는 실제 필요가 생길 때 추가한다.
